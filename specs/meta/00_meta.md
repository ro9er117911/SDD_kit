## 專案背景

### 問題陳述

**現行痛點**:
- 客戶洗錢風險事件分散於多個系統，缺乏整合性摘要
- 人工彙整風險事件耗時且容易遺漏關鍵資訊
- 風險評分標準不一致，依賴人工判斷，缺乏客觀性
- 無法即時掌握高風險客戶動態，影響預警能力
- 監理申報準備作業繁複，資料品質難以保證

### 業務驅動因素

**法規要求**:
- 遵循《洗錢防制法》及相關子法規定
- 滿足金融監理機關（如金管會）對於洗錢防制的監理要求
- 配合國際反洗錢標準（FATF 建議）的持續演進

**風險管理需求**:
- 強化客戶風險分級管理機制
- 建立可量化、可追溯的風險評估模型
- 提升異常交易偵測與預警能力

**營運效率**:
- 降低人工作業成本與錯誤率
- 加速風險決策流程
- 整合分散的風險資訊，提供統一視圖

### 現況分析 (As-Is)

**系統現況**:
- **核心銀行系統**: 儲存客戶基本資料與帳戶資訊，但風險標註功能有限
- **交易監控系統**: 監控可疑交易，但與客戶風險評估未整合
- **KYC 系統**: 客戶盡職調查資料，但無法自動產生風險摘要
- **報表系統**: 定期產出監理申報報表，但缺乏即時性與智能分析

**流程現況**:
1. 各單位分別監控客戶交易與行為
2. 發現異常事件後人工記錄於 Excel 或個別系統
3. 定期（月/季）由法遵人員彙整各類風險事件
4. 人工評估客戶風險等級並撰寫摘要報告
5. 提交管理層審閱與監理機關申報

**痛點總結**:
- 資訊孤島：風險事件分散，缺乏整合
- 時效性差：月/季才彙整，無法即時反應
- 人工密集：大量手動作業，耗時且易錯
- 標準不一：風險評估標準因人而異
- 追溯困難：缺乏完整的評估歷程記錄

### 目標願景 (To-Be)

**系統願景**:
建立智能化的客戶洗錢風險事件摘要模型，能夠：
- 自動整合多個來源系統的風險事件資料
- 運用機器學習演算法進行客戶風險評分
- 即時產生客戶風險摘要與預警通知
- 提供視覺化的風險儀表板與趨勢分析
- 支援監理申報與稽核追蹤需求

**預期成果**:
- 風險事件即時整合與摘要自動化
- 風險評分標準化且可解釋
- 高風險客戶自動預警機制
- 管理層可即時掌握整體風險態勢
- 監理申報效率大幅提升

## 專案範圍

### 範圍內 (In Scope)

**功能範圍**:
- [x] 客戶風險事件資料整合引擎（從多個來源系統擷取）
- [x] 洗錢風險評分模型（基於規則與機器學習）
- [x] 客戶風險事件摘要自動產生功能
- [x] 風險等級分類與標註（高/中/低風險）
- [x] 風險儀表板與視覺化介面
- [x] 風險趨勢分析與統計報表
- [x] 高風險客戶自動預警通知
- [x] 監理申報資料匯出功能
- [x] 稽核追蹤記錄（評分歷程與變更記錄）

**系統範圍**:
- **納入系統**:
  - 核心銀行系統: 客戶基本資料、帳戶資訊
  - 交易監控系統: 可疑交易事件、STR/SAR 資料
  - KYC 系統: 客戶盡職調查資料
  - 外部資料源: 負面新聞、制裁名單、PEP 名單（如適用）
  - 新建 AML 風險摘要系統: 核心模型與介面

**服務範圍**:
- **對象**: 法遵部、風控部、稽核部、管理層
- **地域**: 總行及所有分行（全行適用）
- **時間**: 7x24 運行，每日更新風險評分

### 範圍外 (Out of Scope)

- 不包含實時交易監控功能（由既有交易監控系統負責）
- 不包含客戶盡職調查(KYC)流程管理（由既有 KYC 系統負責）
- 不包含可疑交易申報(STR/SAR)的案件調查流程
- 不包含帳戶凍結或交易阻擋功能
- 不涉及客戶身分驗證(eKYC)機制
- 不包含反洗錢教育訓練管理
- 第一期不包含跨境交易風險分析（未來期程評估）

### 邊界與介面

**上游系統**:
- 核心銀行系統: 提供客戶主檔、帳戶資料（API/批次檔案）
- 交易監控系統: 提供可疑交易事件、警示資料（API/資料庫連結）
- KYC 系統: 提供客戶盡職調查資料、風險分級（API）
- 外部資料服務: 提供負面新聞、制裁名單更新（API/檔案）

**下游系統**:
- 報表系統: 接收風險摘要資料用於監理申報（檔案/API）
- 通知系統: 接收高風險預警通知（Email/訊息推播）
- 稽核系統: 接收風險評估歷程記錄（資料庫介接）

**外部介接**:
- 外部資料供應商 API（如 WorldCheck、Dow Jones 等負面新聞資料庫）
- 金融監理機關申報系統（如適用）

## 變更管理

### 變更控制流程

1. **變更請求**: 填寫變更申請單（包含變更原因、影響範圍、建議方案）
2. **影響評估**: 專案經理與技術負責人評估影響（範圍/時程/成本/風險/品質）
3. **變更審查**: 變更控制委員會審查（依變更等級決定審查層級）
4. **決策**: 核准/退回/延後決定
5. **執行**: 執行變更並更新相關文件（spec.md, plan.md, tasks.md）
6. **驗證**: 確認變更效果符合預期

### 變更控制委員會

| 成員 | 角色 | 權限 |
|------|------|------|
| 專案督導主管 | 主席 | 最終決策權（重大變更） |
| IT 專案經理 | 委員 | 小幅變更決策權、建議權 |
| 法遵部經理 | 委員 | 法規相關變更決策權、建議權 |
| 風控部經理 | 委員 | 模型相關變更決策權、建議權 |
| 技術負責人 | 委員 | 技術可行性評估、建議權 |
