針對 Spec Bot 的規格文件 (spec.md)，提出具體的澄清問題，協助團隊在開發前達成共識：

**技術實作面**
1. **BRD 格式與解析**
   - BRD 是否有固定的 Markdown 模板？
   - 如果 BRD 包含圖片或附件，如何處理？
   - 是否需要支援其他格式（DOCX、PDF）的 BRD？
   - BRD 大小限制為何？（建議 < 100KB）

2. **GPT Prompt 設計**
   - 是否需要針對不同專案類型（後端 API、前端應用、資料管道）使用不同 prompt 模板？
   - GPT 生成的 SDD 品質如何驗證？是否需要自動化評分機制？
   - 如果 GPT 生成的內容不符合預期，重試策略為何？
   - Token 限制：單次 BRD 輸入 token 上限為何？

3. **Docker 沙箱設計**
   - 容器內需要安裝哪些工具？（Git、Node.js、Python、speckit.sh）
   - 容器資源限制？（CPU、Memory、執行時間）
   - 如何處理 speckit.sh 執行失敗？
   - 是否需要保留容器日誌供除錯？

4. **GitHub 整合**
   - GitHub Token 權限範圍？（repo、read:org、workflow）
   - Branch protection rules 如何設定？
   - 是否需要自動指派 reviewers？
   - PR 模板內容是否需要客製化？

5. **錯誤處理與監控**
   - 哪些錯誤需要立即通知 Slack？哪些僅記錄日誌？
   - 是否需要整合 APM 工具（如 Datadog、New Relic）？
   - 日誌保留期限為何？
   - 是否需要錯誤統計儀表板？

**業務流程面**
6. **審核流程**
   - 是否強制要求至少 1 位 SA 審核？
   - 審核 SLA 為何？（建議 24 小時內完成）
   - 如果審核者不在，是否有 escalation 機制？
   - 緊急需求是否有 fast-track 流程？

7. **版本管理**
   - SDD 版本號規則？（建議使用語義化版本 v1.0.0）
   - 如何追溯某個功能是由哪個 BRD 產生？
   - 是否需要支援「比對兩個版本差異」功能？
   - 歷史版本如何歸檔？

8. **多專案支援**
   - 是否需要支援多個 GitHub Repo？
   - 不同專案的 SDD 模板是否不同？
   - 如何避免不同專案的分支命名衝突？
   - 是否需要專案層級的權限控管？

請針對上述問題提供建議答案，或標註為「待團隊討論」，並在 spec.md 中移除 [TBC] 標籤，補充確認後的細節。
